# SwiftSync Architecture

## Package shape

Current `Package.swift` defines three production targets and two test targets:

- `SwiftSync` (public library API + sync engine + reactive wrappers)
- `MacrosImplementation` (macro implementation target)
- `ObjCExceptionCatcher` (Objective-C bridge for `NSException` during `ModelContainer` init)
- `SwiftSyncTests`
- `SwiftSyncMacrosTests`

The package is Swift 6 (`swiftLanguageModes: [.v6]`) with strict concurrency flags enabled on the main library and test targets.

## Main runtime flow

1. `SyncContainer.sync(...)` creates a fresh background `ModelContext`.
2. It forwards work to `SwiftSync.sync(...)` in `API.swift`.
3. `SwiftSync.sync(...)` acquires a lease (actor-serialized per `ModelContainer`) and runs normalize -> match -> apply -> relationship updates -> delete diff -> save.
4. `SyncContainer` observes background `ModelContext.didSave`, processes pending changes on `mainContext`, and posts internal `didSaveChangesNotification`.
5. `@SyncQuery`, `@SyncModel`, and `SyncQueryPublisher` reload from `mainContext` when notified.

## Identity and scoping

- Identity comes from `Model.syncIdentity` (generated by `@Syncable`, or manual conformance).
- Non-parent sync diffs globally by identity.
- Parent sync requires `relationship:` and scopes delete diff to rows in that parent scope.
- Parent-scoped payload sync uses global identity when the identity property is marked `.unique`; otherwise it uses parent-scoped identity keys.

## Payload semantics

`SyncPayload` applies strict semantics used throughout the engine:

- key absent -> ignore
- key present as `NSNull` -> clear/reset when allowed
- key present with value -> apply/coerce when supported

Scalar coercion is broad (`String` <-> numeric/bool/date/uuid/url in supported cases).
Relationship FK linking is strict (`strictValue` path; no cross-type coercion).

## Relationship operations

`SyncRelationshipOperations` controls relationship work per sync call:

- `.insert`
- `.update`
- `.delete`
- `.all`

Generated relationship application from `@Syncable` uses helper functions in `Core.swift`:

- `syncApplyToOneForeignKey`
- `syncApplyToManyForeignKeys`
- `syncApplyToOneNestedObject`
- `syncApplyToManyNestedObjects`

## Reactive read layer

- `@SyncQuery`: reactive list wrapper for SwiftUI.
- `@SyncModel`: reactive detail wrapper for SwiftUI.
- `SyncQueryPublisher`: class-based reactive query for UIKit/plain Swift code.

All three fetch from `syncContainer.mainContext` and rely on `SyncContainer.didSaveChangesNotification` to trigger reloads.

## Export path

- `SyncContainer.export(as:)` fetches rows, sorts by identity string, then calls `exportObject(keyStyle:dateFormatter:)`.
- Relationships are exported inline by default.
- `@NotExport` excludes properties from export.
- Deep key paths (`a.b.c`) are emitted through `exportSetValue`.

## Schema safety and initialization

`SyncContainer(for:...)` validates schema before creating `ModelContainer`:

- Detects many-to-many pairs with zero explicit inverse anchors.
- Throws `SchemaValidationError` with a targeted message.

Initialization uses an Objective-C bridge to convert `NSException` into Swift `Error` (`ObjectiveCInitializationExceptionError`).
Optional `recoverOnFailure` can reset local store files and retry container init.
