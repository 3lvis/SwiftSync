# State Capsule

## Goal

Add basic UIKit support to SwiftSync via `SyncQueryPublisher` and demonstrate it in the demo app with a UIKit "User Tasks" screen.

## Current status

- ✅ Done: Everything — library, tests, demo screen, committed, pushed
- 🔄 In progress: nothing
- ⛔ Blocked by: nothing

## Decisions (don't revisit)

- `SyncQueryPublisher` lives in the `SwiftSync` target alongside `SyncQuery` — no separate target needed
- Class is `@unchecked Sendable` (not `@MainActor`) to match the pattern of the private observers in `ReactiveQuery.swift`; notification closure runs on `queue: .main`
- Shared reload helpers promoted from `private` to `internal` free functions in `ReactiveQuery.swift` — renamed `syncQueryChangedIdentifiers` / `syncQueryChangedModelTypeNames` to avoid collision
- Demo UIKit screen: `UserTasksViewController` (UITableViewController) uses three `SyncQueryPublisher` instances (assignee, reviewer, watcher) merged + deduplicated
- Section identifier for `UITableViewDiffableDataSource` is `String` ("tasks") — enum caused `Sendable` isolation errors with SwiftData imports
- `TaskDetailView` People section: assignee, each reviewer, each watcher are now `NavigationLink`s pushing `UserTasksScreen`

## Constraints

- Must not touch `Core`, `SwiftDataBridge`, `Macros`, `ObjCExceptionCatcher` — they're framework-agnostic and correct
- All tests must pass before any commit
- TDD: tests written first, confirmed failing, then implementation

## Key findings

- Tried: `@MainActor` on `SyncQueryPublisher` + `MainActor.assumeIsolated` in notification closure → `Sending 'notification' risks causing data races`; fixed by dropping `@MainActor` and using `@unchecked Sendable` pattern
- Tried: private `enum Section` inside `UserTasksViewController` for diffable data source → main actor isolation error on `Sendable` constraint; fixed by using `String` as section identifier
- Learned: `PubTask.syncDefaultRefreshModelTypes` includes `PubUser` (via the `assignee` relationship generated by `@Syncable`) — test asserting no reload on `PubUser` change was wrong; fixed by introducing a genuinely unrelated `PubUnrelatedTag` model
