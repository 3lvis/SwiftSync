# Demo: Draft Model Pattern for Export

**Status:** Partially complete — correct pattern now in place; `draft()` macro generation is a future improvement  
**Priority:** Medium — the correctness gap (manual dict surgery bypassing `@RemoteKey`) is fixed

---

## Problem (original)

The demo's update and create flows had two anti-patterns:

### Anti-pattern 1 — Create: insert-upfront (now fixed)

`CreateTaskSheet` inserted a `Task` into `mainContext` immediately at init time,
bound SwiftUI directly to the live persisted object, and required explicit
`delete` + `rollback` on cancel and on failure:

```swift
// Old — insert-upfront, rollback required
syncContainer.mainContext.insert(task)
// ...
func cancelAndDiscard() {
    syncContainer.mainContext.delete(draft)
    syncContainer.mainContext.rollback()
    dismiss()
}
```

### Anti-pattern 2 — Update: manual dict surgery (now fixed)

`EditTaskSheet.save()` built the update body as a raw `[String: Any]` dict,
bypassing `@RemoteKey` / `@RemotePath` mappings entirely:

```swift
// Old — manual dict, @RemoteKey ignored
let body: [String: Any] = [
    "title": draftTitle,
    "description": draftDescription,          // @RemoteKey("description") bypassed
    "state": ["id": draftStateID, "label": draftStateLabel] as [String: Any],
    "assignee_id": draftAssigneeID ?? NSNull()
]
```

If `@RemoteKey` annotations on `Task.descriptionText` or `Task.state` / `Task.stateLabel`
ever changed, the update body would silently diverge from what the macro generates.

---

## Root cause (original doc)

Mutating the live `taskModel` (which lives on `mainContext`) before the API call
left an unsaved dirty mutation on `mainContext`. The background-context path would
then drop conflicting fields on merge. The fix chosen was to revert to manual dict
surgery. The real fix is to never mutate the live persisted object at all.

---

## Current state (after fix/export-draft-pattern)

Both flows now use an **uninserted draft** `Task` and call `exportObject`:

### Create flow (`CreateTaskSheet`)

```swift
// draft is not inserted into any context
let task = Task(id: UUID().uuidString, projectID: projectID, ...)
_draft = State(initialValue: task)

// At save — mutate draft scalars, then export
draft.title = draft.title.trimmingCharacters(in: .whitespacesAndNewlines)
draft.updatedAt = Date()
let body = draft.exportObject(for: syncContainer, relationshipMode: .none)
// On cancel or failure: nothing to clean up — mainContext was never touched
```

### Update flow (`EditTaskSheet`)

```swift
// At save — construct an uninserted Task from @State draft vars, then export
let draft = Task(
    id: taskID,
    projectID: projectID,
    assigneeID: draftAssigneeID,
    authorID: authorID,
    title: draftTitle.trimmingCharacters(in: .whitespacesAndNewlines),
    descriptionText: draftDescription,
    state: draftStateID,
    stateLabel: draftStateLabel,
    createdAt: createdAt,
    updatedAt: Date()
)
let body = draft.exportObject(for: syncContainer, relationshipMode: .none)
```

`@RemoteKey("description")`, `@RemoteKey("state.id")`, `@RemoteKey("state.label")` are
now applied correctly by the macro-generated `exportObject`. The server's `updateTask`
handler is designed to accept the full exported object (immutable fields like `id`,
`project_id`, `author_id`, `created_at` are validated for consistency but not mutated).

---

## Key finding: uninserted `@Model` objects are safe for `exportObject`

`@Model` objects that have never been inserted into a `ModelContext` have their
properties backed by a heap-allocated `DefaultBackingData` buffer initialized at
`init` time. Property reads and writes are safe. `apply()` and `exportObject` work
correctly on uninserted objects (verified by existing tests in `SyncTests.swift`).

---

## Remaining boilerplate concern

The update flow constructs a `Task(...)` by listing every scalar field at the call
site. For a model with many fields this is error-prone: adding a new field requires
updating every draft construction site.

### Future improvement: `@Syncable` generates `draft()` method

The `@Syncable` macro already knows all scalar fields. It should generate a `draft()`
method that returns a new uninserted copy of the object with all scalar fields copied
and relationships set to their defaults (`nil` / `[]`):

```swift
// Generated by @Syncable:
func draft() -> Task {
    Task(
        id: self.id,
        projectID: self.projectID,
        assigneeID: self.assigneeID,
        authorID: self.authorID,
        title: self.title,
        descriptionText: self.descriptionText,
        state: self.state,
        stateLabel: self.stateLabel,
        createdAt: self.createdAt,
        updatedAt: self.updatedAt
    )
    // not inserted into any context
}
```

Call sites would then become:

```swift
// EditTaskSheet — no field listing at call site
let draft = taskModel.draft()
draft.title = draftTitle.trimmingCharacters(in: .whitespacesAndNewlines)
draft.descriptionText = draftDescription
draft.state = draftStateID
draft.stateLabel = draftStateLabel
draft.assigneeID = draftAssigneeID
draft.updatedAt = Date()
let body = draft.exportObject(for: syncContainer, relationshipMode: .none)
```

This would also allow the SwiftUI form to bind directly to `$draft.title`,
`$draft.descriptionText`, etc., replacing the separate `@State` scalar vars.
