# SwiftData Dirty-Tracking Gap in To-Many Relationships

**Status:** Fixed on `main` (commit `346f048`)  
**Affects:** Persistent (on-disk) SQLite stores only — in-memory stores are not affected  
**Risk if regressed:** `@SyncModel` / `@SyncQuery` serve stale relationship data until the next background poll

---

## Background

SwiftSync's reactive update pipeline works like this:

```
background ModelContext.save()
    ↓
SyncContainer.modelContextDidSave(_:)         // observes NSManagedObjectContextDidSave
    → collects changedIDs from notification
    → faults each ID into mainContext
    → posts SyncContainer.didSaveChangesNotification
        ↓
SyncModelObserver / SyncQueryObserver
    → shouldReload() checks changedTypeNames ∩ observedTypeNames
    → reload() re-fetches from mainContext
    → @Published var model/rows fires
        ↓
SwiftUI re-renders the view
```

The entire chain depends on the owning model's `PersistentIdentifier` appearing in the `NSManagedObjectContextDidSave` notification's `updatedObjects`. If it is absent, `modelContextDidSave` never faults the model into `mainContext`, never posts the type name in `changedModelTypeNames`, and the observer never reloads.

---

## The Problem

SwiftData (built on Core Data) only marks a model's **own store row** dirty when a **scalar column** on that model changes. Writing to a to-many relationship key path only dirties the **join-table rows** — not the owning row.

This means: after `syncApplyToManyForeignKeys` writes a new membership set (e.g., adding a reviewer to a task), the Task's store row is untouched. The Task's `PersistentIdentifier` is absent from `NSManagedObjectContextDidSave`'s `updatedObjects`. `modelContextDidSave` never sees the Task. The notification fires with only join-table type names. The `SyncModelObserver<Task>` gets no notification it can act on. The `@SyncModel` serving `TaskDetailView` is never told to reload.

The UI stays stale until the next background poll (14s in the Demo).

### Why it only manifests on persistent stores

In-memory SQLite stores behave differently — SwiftData always surfaces the owning row's identifier in the save notification regardless of whether a scalar changed. This is why:

- The unit tests (`testToManyRelationshipOnlySyncUpdatesMainContextImmediately`) pass even without the fix — the assertion `notificationIDs.contains(taskID)` is always true in-memory
- The bug manifests on device and on the Simulator, both of which use the Demo's persistent on-disk store (`ApplicationSupport/SwiftSyncDemo/client-cache.store`)

### Affected relationship pattern

The gap only affects models that use **to-many relationships with no explicit `@Relationship` inverse anchor** — which is the exact pattern used in the Demo for `Task.reviewers` and `Task.watchers`:

```swift
// Task has reviewers and watchers with no declared inverse on User
var reviewers: [User]
var watchers: [User]
```

Models with a proper bidirectional `@Relationship(inverse:)` declaration may not trigger this gap because Core Data can track the change through the inverse side, but this is not guaranteed and should not be relied upon.

## The Fix (commit `346f048`)

### Protocol addition — `SyncUpdatableModel.syncMarkChanged()`

```swift
// Core.swift
public protocol SyncUpdatableModel: SyncModelable {
    // ...existing requirements...

    /// Forces a no-op write on a scalar property so SwiftData marks the model's
    /// persistent store row as dirty. This guarantees the model's PersistentIdentifier
    /// surfaces in ModelContext.didSave's updatedObjects after a to-many relationship
    /// change — necessary because SwiftData only dirties the owning row when a scalar
    /// column changes, not when only join-table rows change.
    func syncMarkChanged()
}

public extension SyncUpdatableModel {
    func syncMarkChanged() {}  // default no-op for hand-written conformances
}
```

### Macro-generated implementation — `@Syncable`

`SyncableMacro` generates a real implementation for every `@Syncable` model by performing a no-op write on the identity property:

```swift
// Generated by @Syncable
func syncMarkChanged() {
    self.id = self.id   // no-op scalar write — forces Core Data to mark the row dirty
}
```

### Call site — `syncApplyToManyForeignKeys`

Called in both membership-change paths — when clearing the set (null payload) and when writing a new set:

```swift
if modelIDSet(current) != modelIDSet(next) {
    owner[keyPath: relationship] = next
    owner.syncMarkChanged()   // ← forces dirty bit on owning row
    return true
}
```

### Constraint tightening

The `SyncModelable` overload of `syncApplyToManyForeignKeys` was constrained from `<Owner>` to `<Owner: SyncUpdatableModel>` — all real callers already satisfy this, and it makes the `syncMarkChanged()` call statically guaranteed.

## Test coverage

`SyncRelationshipIntegrityTests.testToManyRelationshipOnlySyncUpdatesMainContextImmediately` pins the contract:

- Seeds a task with no tags, then syncs a payload that adds two tags (relationship-only change, no scalar change)
- Captures the `SyncContainer.didSaveChangesNotification` emitted during the sync
- Asserts the task's `PersistentIdentifier` is in the notification's `changedIDs`
- Asserts the relationship data is correct in `mainContext` after the sync

**Limitation:** the assertion `notificationIDs.contains(taskID)` passes even without `syncMarkChanged` when using an in-memory store — in-memory stores always surface the owner. The test cannot be driven red in unit tests. It is kept as a non-regression anchor and contract documentation, not as a red/green proof of the fix.

## Investigation history

This issue was originally identified during investigation of a different bug: reviewers/watchers not updating instantly in the Demo's edit modal after saving. The instrumentation was added on branch `debug/reviewer-watcher-stale-ui`, branching from commit `85b7fa0` — which is before the `syncMarkChanged` fix (`346f048`) landed on `main`.

Despite `syncMarkChanged` being absent on that branch, the logs showed `Demo.Task` in `changedTypeNames` after every relationship save and `SyncModelObserver` reloading correctly every time. This is unexplained: the Demo uses a persistent on-disk SQLite store (`ApplicationSupport/SwiftSyncDemo/client-cache.store`), so the dirty-tracking gap should have manifested. The most likely explanations are (a) a Simulator-specific SQLite behavior that always surfaces the owning row, or (b) an incidental scalar write elsewhere in the sync path that happened to touch the Task row. It was not investigated further because the gap was already fixed on `main`.

The actual root cause of the visible bug turned out to be a different ordering issue in `DemoSyncEngine`: `syncProjectTasksInternal` was running *after* `syncTaskDetailInternal`, writing a stale project-list snapshot (which contained the pre-save relationship membership) on top of the correct data that `syncTaskDetailInternal` had just written. Fixed by swapping the order so `syncTaskDetailInternal` always runs last (commit `3519db9` on `debug/reviewer-watcher-stale-ui`).

The `syncMarkChanged` fix (`346f048`) is correct and necessary regardless — it guards against a real SwiftData behavior that the investigation happened not to trigger during that session.

## Open work / future considerations

### 1. The test cannot be driven red — consider a persistent-store integration test

The existing unit test uses an in-memory store and cannot fail even without the fix. A proper regression test would require a persistent (file-backed) SQLite store. This is possible but requires test teardown to clean up the on-disk file.

**Options:**
- Add an `XCTestCase` subclass that creates a `ModelConfiguration` with a temp file URL and deletes it in `tearDown`
- Use a persistent store only for the specific test(s) that need to verify dirty-tracking behavior
- Accept the current documentation-only test as sufficient given the fix is already in `syncApplyToManyForeignKeys`

### 2. Hand-written `SyncUpdatableModel` conformances get a no-op `syncMarkChanged()`

The default implementation is a no-op, which means any model that manually conforms to `SyncUpdatableModel` (rather than using `@Syncable`) will silently not dirty its row after a to-many relationship change. This is the correct default for conformances that don't need it, but could be a footgun for conformances that do.

**Options:**
- Document this clearly in the protocol's doc comment (already done)
- Add a warning/diagnostic in the macro if `@Syncable` is applied to a model that also manually implements `syncMarkChanged` with a no-op body
- Accept the current behavior — real-world models should all use `@Syncable`

### 3. The `syncApplyToManyForeignKeys<Owner: PersistentModel>` overload does not call `syncMarkChanged()`

There are two overloads of `syncApplyToManyForeignKeys`:

```swift
// Overload 1 — Owner is plain PersistentModel (no syncMarkChanged available)
public func syncApplyToManyForeignKeys<Owner, Related: PersistentModel>(...)

// Overload 2 — Owner: SyncUpdatableModel (calls syncMarkChanged)
public func syncApplyToManyForeignKeys<Owner: SyncUpdatableModel, Related: SyncModelable>(...)
```

Overload 1 is used when `Related` is a plain `PersistentModel` (not `SyncModelable`). It cannot call `syncMarkChanged()` because `Owner` is not constrained to `SyncUpdatableModel`. In practice, all real Demo models use Overload 2, but if a future caller ends up on Overload 1, the dirty-tracking gap would silently return.

**Options:**
- Constrain Overload 1's `Owner` to `SyncUpdatableModel` as well (may break callers that don't conform)
- Add a note in Overload 1's doc comment warning that `syncMarkChanged` is not called
- Remove Overload 1 if it has no real callers

### 4. Confirm the dirty-tracking gap is reproducible without the fix

The Demo uses a persistent on-disk store, so the gap should be reproducible on device and Simulator. However, during the `debug/reviewer-watcher-stale-ui` investigation, the gap was not observed even though `syncMarkChanged` was absent on that branch. The reason is unknown.

A focused test would be: temporarily revert `syncMarkChanged` from `syncApplyToManyForeignKeys` on a local branch, run the Demo on device with the `BugDebug` instrumentation re-enabled, and confirm `Demo.Task` is absent from `changedTypeNames` after a relationship-only save. This would close the question of whether the fix is actively necessary or merely defensive.
