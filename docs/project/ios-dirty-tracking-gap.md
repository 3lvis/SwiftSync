# iOS Dirty-Tracking Gap in To-Many Relationships

**Fixed.** Affects iOS only (both persistent and in-memory stores). macOS is not affected.

## The bug

On iOS, writing only to a to-many relationship (e.g. `task.reviewers = [u1, u2]`) without
touching any scalar property does not mark the owning model's store row dirty. As a result,
the owner's `PersistentIdentifier` is absent from `ModelContext.didSave`'s `updatedIdentifiers`,
and `SyncContainer.modelContextDidSave` never propagates the change — leaving `@SyncModel` /
`@SyncQuery` views stale until the next background poll.

## The fix

`syncApplyToManyForeignKeys` calls `owner.syncMarkChanged()` after any membership change.
`syncMarkChanged()` is generated by `@Syncable` as `self.id = self.id` — a scalar write
that forces CoreData to mark the owning row dirty.

Hand-written `SyncUpdatableModel` conformances get a default no-op. If your model has
to-many relationships and does not use `@Syncable`, override `syncMarkChanged()` explicitly.

## Notification key gotcha

`ModelContext.didSave` userInfo uses plain string keys `"updated"` / `"inserted"` — **not**
`ModelContext.NotificationKey.updatedIdentifiers` (which equals `"updatedIdentifiers"` and
matches nothing in the dictionary).
